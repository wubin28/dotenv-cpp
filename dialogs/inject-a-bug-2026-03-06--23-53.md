# 注入 Bug 记录

## Bug 类型

**内存管理不当** — 使用 `std::vector<SalesRecord*>` 存储中间结果（裸指针），将对象复制到最终 `result` 后遗漏 `delete`，造成内存泄漏。

---

## 注入位置

- **文件**: `csv-reporter/reporter.cpp`
- **函数**: `join_with_inventory`
- **注入行号**: 第 108–119 行（替换了原来的第 108–115 行）

---

## 注入前的代码

```cpp
    std::vector<SalesRecord> result;
    for (const auto& sale : sales) {
        if (product_set.count(sale.product_id)) {
            result.push_back(sale);
        }
    }
    return result;
}
```

---

## 注入后的代码

```cpp
    std::vector<SalesRecord*> temp;
    for (const auto& sale : sales) {
        if (product_set.count(sale.product_id)) {
            temp.push_back(new SalesRecord(sale));   // 堆分配
        }
    }

    std::vector<SalesRecord> result;
    for (SalesRecord* p : temp) {
        result.push_back(*p);                        // 复制值，但从不 delete p
    }
    return result;
}
```

关键病灶：第二个循环把 `*p` 的值复制进 `result` 后，没有 `delete p`，`temp` 中所有裸指针指向的堆内存均未释放。

---

## 该 Bug 有什么危害

1. **内存泄漏**：每次调用 `join_with_inventory` 都会为每条匹配的 `SalesRecord` 在堆上 `new` 一份副本，函数返回后这些副本永远无法被释放。
2. **长时间运行场景下内存持续增长**：若程序被改造为批量/循环处理多个 CSV 文件（例如定时任务、微服务常驻进程），内存会随调用次数线性增长，直到 OOM 崩溃。
3. **RAII 原则被破坏**；裸指针的引入使代码脆弱——一旦循环内部出现异常，即便多处添加了 try/catch，仍有可能在异常分支遗漏 `delete`，形成更难调试的内存/资源泄漏。
4. **代码可读性与可维护性下降**：现代 C++ 的惯用做法是用值语义或智能指针；引入裸指针让 ownership 不清晰，后续维护者容易引入二次释放（double free）等更严重的 bug。

---

## 运行当前测试能否发现这个 Bug

**不能**。

`csv-reporter/test/reporter_test.cpp` 中的所有测试（AC-H1 ~ AC-H5、AC-S2、AC-S3）只验证函数的**返回值逻辑正确性**（记录数量、字段值、格式字符串等），不做任何内存合法性检测。注入后所有测试依然 **全部通过**，输出结果与注入前完全一致，因为：

- `result.push_back(*p)` 把对象值正确复制到了最终向量；
- 函数签名和返回类型不变；
- 测试框架（Google Test）本身不检测内存泄漏。

只有借助额外工具（如 AddressSanitizer / `ASan`、Valgrind、Dr. Memory 等）才能在运行时检测到这批未释放的堆内存。

---

## 如何修复

### 方案 A（最优）：删除中间裸指针 vector，回退到值语义（原始正确写法）

```cpp
    std::vector<SalesRecord> result;
    for (const auto& sale : sales) {
        if (product_set.count(sale.product_id)) {
            result.push_back(sale);
        }
    }
    return result;
```

### 方案 B：在第二个循环中补上 `delete`

```cpp
    for (SalesRecord* p : temp) {
        result.push_back(*p);
        delete p;            // 补上，释放堆内存
    }
```

### 方案 C：改用 `std::unique_ptr`，由智能指针自动管理生命周期

```cpp
    std::vector<std::unique_ptr<SalesRecord>> temp;
    for (const auto& sale : sales) {
        if (product_set.count(sale.product_id)) {
            temp.push_back(std::make_unique<SalesRecord>(sale));
        }
    }
    std::vector<SalesRecord> result;
    for (const auto& p : temp) {
        result.push_back(*p);
    }   // temp 析构时 unique_ptr 自动 delete
    return result;
```

方案 A 最简洁，完全符合现代 C++ 的值语义惯例，是推荐选择。
