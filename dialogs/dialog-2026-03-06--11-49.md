## 需求一：本地文件批量加密/签名工具（`file-vault`）

### 功能描述

用 dotenv 管理加密密钥（`SECRET_KEY`）、算法参数（`CIPHER_MODE`）、输入目录（`INPUT_DIR`）和输出目录（`OUTPUT_DIR`），在终端对指定目录下的文件批量加密或验签。

### 人类程序员容易犯的 C++ Bug

| Bug 类型 | 具体表现 |
|---|---|
| **空指针崩溃** | 用 `std::getenv("SECRET_KEY")` 而非 `dotenv::getenv()`，返回 `nullptr` 后直接 `strlen()` 崩溃 |
| **堆缓冲区溢出** | `new char[file_size]` 读取加密输出时，密文比明文长（如 AES padding），写出界 |
| **内存泄漏** | `char* buf = new char[n]` 后遇到文件打开失败提前 `return`，`buf` 未 `delete[]` |
| **字符串转换未处理异常** | `int key_len = std::stoi(dotenv::getenv("KEY_LENGTH",""))` 当值为空字符串时 `stoi` 抛 `std::invalid_argument`，程序崩溃 |
| **性能问题** | 每处理一个文件都在循环内调用 `dotenv::getenv()`（封装了 `std::getenv` 系统调用），而非在 main 里一次性读取缓存到局部变量 |

### 特点

- 直接对应 dotenv 的核心价值：**敏感密钥（密钥、IV）与代码分离**
- 文件处理涉及 `new`/`delete[]`、文件句柄，错误路径多
- 功能点单一，学员能快速理解业务逻辑，专注于 bug 修复

### 优势

- **Bug 密度高且真实**：空指针、堆溢出、内存泄漏可以在同一份代码里共存，学习价值密集
- **可感知的错误后果**：崩溃、数据损坏、加密文件异常，结果直观
- **dotenv 使用最自然**：密钥管理天然需要 `.env` 与代码分离

### 劣势

- 需要理解基础加密算法（XOR/AES）或使用 OpenSSL，增加了前置知识门槛
- Windows 上文件路径（`\` vs `/`）可能分散注意力

### 适用场景

适合**有一定 C++ 基础、已学过指针和动态内存**的学员。重点练习：AI 辅助做代码审查（Code Review）、自动检测内存泄漏（Valgrind/AddressSanitizer 辅助分析）。

## 需求二：系统健康监控与告警 CLI 工具（`sys-watchdog`）

### 功能描述

用 dotenv 管理监控阈值（`CPU_THRESHOLD=80`）、内存上限（`MEM_LIMIT_MB=1024`）、告警日志路径（`LOG_FILE`）、轮询间隔（`POLL_INTERVAL_SEC=5`），周期性采集 CPU/内存使用率，超阈值时写入告警日志，通过终端实时显示仪表盘。

### 人类程序员容易犯的 C++ Bug

| Bug 类型 | 具体表现 |
|---|---|
| **整数溢出** | 内存字节数用 `int` 存储（最大约 2GB），在 8GB+ 内存的机器上计算出负值 |
| **字符串转数字未防御** | `int threshold = std::stoi(dotenv::getenv("CPU_THRESHOLD",""))` 当 `.env` 里填了 `80%`（带百分号）时 `stoi` 抛异常 |
| **悬空指针** | 将 `dotenv::getenv(...).c_str()` 的返回值存入 `const char* path`，`std::string` 临时对象销毁后 `path` 成为悬空指针 |
| **性能热路径问题** | 在 while 轮询循环内每次都调用 `dotenv::getenv()` 读取阈值（系统调用），应在循环外缓存到 `const int threshold` |
| **日志文件句柄泄漏** | `std::ofstream log(path)` 在循环内每次都打开/关闭而非保持句柄，遇到磁盘满时异常，句柄未关闭 |

### 特点

- 体现了 dotenv 的**多环境配置切换**价值（本地 `.env` vs 生产 `.env.prod`）
- 数值类型转换是最核心的陷阱，与 dotenv **"所有值都是字符串"的劣势**直接挂钩
- 有持续运行的 `while` 循环，性能问题可以用实际延迟数据量化

### 优势

- **Bug 与 dotenv 强耦合**：整数溢出、转换失败都是因为 dotenv 只返回字符串，强化学员对库局限性的认知
- **可测量的性能差异**：轮询循环内外调用 `getenv()` 的性能差异可以用计时器量化，AI 辅助优化的效果可见
- **业务逻辑学员熟悉**：CPU/内存监控是大家日常接触的概念，不需要额外解释业务背景

### 劣势

- Windows 和 Linux 获取系统资源的 API 不同（`GlobalMemoryStatusEx` vs `/proc/meminfo`），跨平台代码量多
- 需要一定的系统编程知识

### 适用场景

适合**学习 AI 辅助性能优化和代码重构**的场景。重点练习：让 AI 识别热路径上的低效调用、自动生成类型安全的包装函数（如 `getenv_int()`）、生成边界值测试用例。

## 需求三：多源 CSV 数据聚合报表工具（`csv-reporter`）

### 功能描述

用 dotenv 管理多个 CSV 文件路径（`SALES_FILE`、`INVENTORY_FILE`）、分隔符（`DELIMITER=,`）、过滤条件（`MIN_AMOUNT=1000`）、报表格式（`OUTPUT_FORMAT=json`），在终端读取并 Join/Filter 多个 CSV，输出统计摘要。

### 人类程序员容易犯的 C++ Bug

| Bug 类型 | 具体表现 |
|---|---|
| **越界访问（最常见）** | 解析 CSV 行后 `fields[3]` 访问第 4 列，当某行只有 2 列时 `std::vector::operator[]` 越界，UB |
| **内存管理不当** | 使用 `std::vector<std::string*>` 存储所有行（原始指针），Join 结束后 `for` 循环逐个 `delete`，但异常分支遗漏 `delete` |
| **字符串拼接性能问题** | 在循环里用 `result_str += row_str + ","` 构建 JSON，N 行数据产生 O(N²) 次复制，应使用 `std::ostringstream` |
| **大文件内存耗尽** | 将百万行 CSV 全量读入 `std::vector<std::string>`，内存耗尽崩溃，应流式处理 |
| **浮点精度问题** | `double total = std::stod(dotenv::getenv("MIN_AMOUNT","0"))` 与 CSV 中读出的浮点数直接 `==` 比较，结果不稳定 |

### 特点

- 完全是纯业务逻辑，**无平台差异**，学员专注于 C++ 数据处理 bug
- 越界访问、性能问题、内存问题在同一功能里**自然且密集地共存**
- CSV 文件可以由脚本或 AI 生成任意规模的测试数据，便于验证性能优化效果

### 优势

- **零平台依赖**：在 Windows/Linux/macOS 上行为完全一致，不会因系统 API 差异分散注意力
- **难度可分级**：从 10 行 CSV（功能验证）到 100 万行（性能调优），同一段代码可以练习不同深度的问题
- **AI 辅助空间最大**：代码审查、重构建议、自动生成测试数据、性能分析都有明确的落地点；`vector<string*>` → `vector<string>` 这类重构是 AI 辅助代码改进的经典案例

### 劣势

- CSV 解析本身的边界情况多（带引号的字段、换行符在字段内），可能消耗过多讨论时间
- 和 dotenv 的核心价值关联相对间接（dotenv 只提供路径和参数，不是主角）

### 适用场景

适合**各难度层次的学员**，是三者中**最推荐作为培训实战案例**的选择。重点练习：AI 辅助代码审查（识别越界、内存泄漏）、AI 辅助性能 profiling 与重构（`string+` → `ostringstream`）、TDD（先让 AI 生成测试用例，再让学员修复 bug）。
