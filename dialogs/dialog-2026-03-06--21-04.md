# Understand the Code Skill

Help a C++ developer who just inherited a project quickly understand it by producing a thorough, copy-paste-ready analysis document.

## Trigger

Use this skill when the user says any of:
- "我刚接手这个项目"
- "I just inherited / took over this project"
- "help me understand this codebase"
- or invokes it explicitly with `/understand-the-code-by-create-skill`

---

## Workflow

### Step 1 — Read Key Project Files

Read the following files **in order**, stopping only when you have enough information to answer all three questions below. Do not skip files that exist.

| Priority | File(s) | Purpose |
|----------|---------|---------|
| 1 | `README.md` | Overview, features, installation, usage |
| 2 | `CMakeLists.txt` (root) | Build targets, test setup, dependencies |
| 3 | `include/**/*.h` | API surface, key types |
| 4 | `tests/*.cc`, `tests/*.cpp` | Test organization and fixtures |
| 5 | `example.cpp` (or similar) | End-to-end usage example |
| 6 | `build/CMakeCache.txt` | Already-configured generators and options |

---

### Step 2 — Answer the Three Questions

Structure your answer around these three questions. Each answer must be **specific enough that the reader can follow it without guessing**.

---

#### Q1 — Project Overview (功能、特点、优势、劣势、适用场景)

- Main functionality in one concise paragraph
- Key technical characteristics (header-only? external dependencies? OS compatibility?)
- Advantages and disadvantages (be honest about limitations)
- Recommended use cases and anti-use-cases

---

#### Q2 — Build and Run on Windows 11 + VS2022 + PowerShell

Cover every sub-step needed, in order:

1. **Prerequisites** — list required tools with minimum versions (CMake, Visual Studio workload name, Git, etc.)
2. **Clone / open the project** — exact command or path
3. **Configure** — exact `cmake` configure command (generator, build dir, any required `-D` flags)
4. **Build** — exact `cmake --build` command, specifying `--config Debug` or `Release` as appropriate
5. **Sample data files** — if the project needs a `.env`, config file, or other input file at runtime, provide the exact file path and realistic sample content the user can copy
6. **Run** — exact command to launch the compiled binary, including any required arguments or working directory
7. **Expected output** — show what a successful run looks like (truncated is fine; highlight the key lines)

Write each step as a PowerShell code block so it is copy-paste ready. Warn about any Windows-specific pitfalls (path separators, `Debug\` vs `Release\` subdirectory, `_putenv_s` vs `setenv`, etc.).

---

#### Q3 — Run Automated Tests

- Exact CTest or test-runner command (with `--config Debug` or `--config Release` as appropriate)
- Expected output when all tests pass (number of tests, "Passed" summary)
- How to run a single test if available

---

### Step 3 — Verify Completeness

Before saving, self-check:
- [ ] Every PowerShell command is a complete, runnable line — no placeholders like `<your-path>`
- [ ] Any required data files (e.g., `.env`) have sample content provided
- [ ] Windows-specific flags and directory separators are used throughout
- [ ] Both "fresh build" and "incremental build" paths are mentioned where meaningful
- [ ] Common failure modes are called out (missing `.env`, wrong config, PATH issues)

---

### Step 4 — Save as Timestamped File

Save the complete answer to:

```
dialogs/understand-the-code-<timestamp>.md
```

where `<timestamp>` follows the format `YYYY-MM-DD--HH-MM` (e.g., `2026-03-06--20-15`), based on current local time. Create the `dialogs/` directory if it does not exist.
