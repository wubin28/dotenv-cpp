## Review: reporter.cpp

### Summary
The file is well-structured and readable. Most functions are correct and idiomatic C++17. However, it contains **one injected critical bug**: a memory leak in `join_with_inventory` where every heap-allocated `SalesRecord` is copied out but never freed. The same function also carries a performance issue as a direct consequence of the bug. The rest of the file is clean.

---

### Critical Issues

**[C1] Memory leak — every matched `SalesRecord` is leaked** (reporter.cpp)

```cpp
std::vector<SalesRecord*> temp;
for (const auto& sale : sales) {
    if (product_set.count(sale.product_id)) {
        temp.push_back(new SalesRecord(sale));  // heap-allocated …
    }
}

std::vector<SalesRecord> result;
for (SalesRecord* p : temp) {
    result.push_back(*p);   // … copied by value, but p is NEVER deleted
}
return result;
```

**Root cause**: Each matching `SalesRecord` is allocated with `new` into a raw-pointer vector. The objects are then *copied* into `result`, but the raw pointers in `temp` are never passed to `delete`. The destructor of `std::vector<SalesRecord*>` destroys the pointers themselves (the 8-byte address), not the pointed-to objects. Every call to `join_with_inventory` leaks one `sizeof(SalesRecord)` per matching row.

**Fix**: Eliminate the `temp` vector entirely and push by value directly:

```cpp
std::vector<SalesRecord> join_with_inventory(
    const std::vector<SalesRecord>& sales,
    const std::vector<InventoryRecord>& inventory)
{
    std::unordered_set<std::string> product_set;
    for (const auto& inv : inventory) {
        product_set.insert(inv.product_id);
    }

    std::vector<SalesRecord> result;
    for (const auto& sale : sales) {
        if (product_set.count(sale.product_id)) {
            result.push_back(sale);   // value copy — no heap allocation needed
        }
    }
    return result;
}
```

### Medium Priority Issues

**[M1] Unnecessary heap allocation in `join_with_inventory`** (reporter.cpp)

This is the performance face of C1. Even if the leak were patched with `delete` in the second loop, allocating each record on the heap and then copying it is pure overhead. `SalesRecord` is a value type; there is no reason to ever put it on the heap here. The fix in C1 resolves this simultaneously.

---

### Low Priority Issues

**[L1] Missing `reserve()` before bulk push_back** (reporter.cpp and similar)

Functions like `join_with_inventory`, `parse_sales_rows`, and `parse_inventory_rows` all push into a `result` vector without a `reserve()`. For large CSV files this causes repeated reallocation and copying. A conservative `result.reserve(sales.size())` (or `all_rows.size()`) before the loop avoids this.
