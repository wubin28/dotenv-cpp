## Quick Checklist

### Memory Management
- [ ] Every `new` / `new[]` has a matching `delete` / `delete[]` on **all** control-flow paths
- [ ] Raw-pointer containers (`vector<T*>`) release every element before the container goes out of scope
- [ ] Prefer value types or smart pointers (`unique_ptr`, `shared_ptr`) over raw owning pointers
- [ ] RAII: file handles, streams, and other resources are tied to object lifetime
- [ ] No use-after-free, double-free, or dangling-pointer dereferences

### Undefined Behaviour
- [ ] `std::isspace` / `std::isalpha` / `std::isupper` etc. receive `unsigned char`, not plain `char` or `int` cast from `char`

### Performance
- [ ] No unnecessary heap allocations where stack/value semantics suffice
- [ ] Large objects passed by `const&` not by value
- [ ] `std::vector` pre-sized with `reserve()` before bulk insertions
- [ ] Inner joins / lookups use O(1) `unordered_set` / `unordered_map` (not O(n) linear scan)

## 1. Reporter: Memory Leak in `join_with_inventory`

**File**: `csv-reporter/reporter.cpp`, function `join_with_inventory`  
**Severity**: Critical / High (injected memory leak bug)

### What the code does

```cpp
std::vector<SalesRecord*> temp;
for (const auto& sale : sales) {
    if (product_set.count(sale.product_id)) {
        temp.push_back(new SalesRecord(sale));   // heap-allocated
    }
}

std::vector<SalesRecord> result;
for (SalesRecord* p : temp) {
    result.push_back(*p);   // copied — but p is NEVER deleted
}
return result;
```

Every `SalesRecord` that passes the inventory filter is heap-allocated with `new`. The objects are copied into `result` but the raw pointers in `temp` are **never freed**. Every call to `join_with_inventory` leaks one `SalesRecord` object per matching row.

### Fix

Remove the `temp` vector entirely. Push matched records directly into `result` by value:

```cpp
std::vector<SalesRecord> join_with_inventory(
    const std::vector<SalesRecord>& sales,
    const std::vector<InventoryRecord>& inventory)
{
    std::unordered_set<std::string> product_set;
    for (const auto& inv : inventory) {
        product_set.insert(inv.product_id);
    }

    std::vector<SalesRecord> result;
    for (const auto& sale : sales) {
        if (product_set.count(sale.product_id)) {
            result.push_back(sale);   // no heap allocation needed
        }
    }
    return result;
}
```

## 2. dotenv.h: UB in `ltrim` / `rtrim` lambdas

**File**: `include/laserpants/dotenv/dotenv.h`, functions `ltrim` and `rtrim`  
**Severity**: High (undefined behaviour on non-ASCII characters in `.env` values)

### What the code does

```cpp
inline void dotenv::ltrim(std::string& s) {
    s.erase(s.begin(), std::find_if(s.begin(), s.end(),
        [](int c) { return !std::isspace(c); }));   // BUG: 'int c' from signed char
}
inline void dotenv::rtrim(std::string& s) {
    s.erase(std::find_if(s.rbegin(), s.rend(),
        [](int c) { return !std::isspace(c); }).base(), s.end());  // same bug
}
```

On platforms where `char` is signed (x86/x64 MSVC, GCC, Clang defaults), characters with values 128–255 are represented as negative `int` values (-128 to -1) when widened. The C standard requires that `std::isspace` receives a value representable as `unsigned char` or `EOF`; passing a negative value other than `EOF` is **undefined behaviour**.

### Fix

Cast to `unsigned char` inside the lambda:

```cpp
inline void dotenv::ltrim(std::string& s) {
    s.erase(s.begin(), std::find_if(s.begin(), s.end(),
        [](unsigned char c) { return !std::isspace(c); }));
}
inline void dotenv::rtrim(std::string& s) {
    s.erase(std::find_if(s.rbegin(), s.rend(),
        [](unsigned char c) { return !std::isspace(c); }).base(), s.end());
}
```

### Detection

```powershell

## 3. example.cpp: `std::getenv` null-pointer dereference

**File**: `example.cpp`  
**Severity**: High (crashes when `.env` file is missing or variables are not set)

### What the code does

```cpp
std::cout << std::getenv("DATABASE_USERNAME") << std::endl;

const char* user = std::getenv("DATABASE_USERNAME");

## 5. dotenv.h: Redundant `find` Calls in `find_var_start`

**File**: `include/laserpants/dotenv/dotenv.h`, function `find_var_start`  
**Severity**: Medium (minor performance — two passes over the string instead of one)

### What the code does

```cpp
size_t p1 = str.find('$', pos);      // finds first '$'
size_t p2 = str.find("${", pos);     // finds first '${'
size_t pos_var = (std::min)(p1, p2);
```

Every `${` occurrence also satisfies the single-char `find('$')`, so `p1 <= p2` always. The `std::min` always picks `p1`. The second `find` is used only to detect whether the matched `$` is followed by `{`. This can be done with a single find:

### Fix

```cpp
inline size_t dotenv::find_var_start(const std::string& str, size_t pos,
                                     std::string& start_tag)
{
    size_t pos_var = str.find('$', pos);
    if (pos_var != std::string::npos) {
        start_tag = (pos_var + 1 < str.size() && str[pos_var + 1] == '{')
                    ? "${" : "$";
    }
    return pos_var;
}
```

## 1. Raw Owning Pointer Anti-patterns

### Pattern: Vector of raw owning pointers

```cpp
// BAD — ownership unclear, leak-prone
std::vector<T*> items;
items.push_back(new T(value));
// ... if items is never explicitly freed, all T objects leak
```

**Why it leaks**: `std::vector<T*>` destructs the pointer values but does NOT call `delete` on each pointer. You must iterate and delete manually — or, better, avoid raw owning pointers entirely.

**Fix options (prefer in this order)**:

| Option | When to use |
|--------|-------------|
| `std::vector<T>` (value) | Always prefer when T is copyable/movable and size is bounded |
| `std::vector<std::unique_ptr<T>>` | When T is non-copyable or polymorphic |
| `std::vector<std::shared_ptr<T>>` | When shared ownership is truly needed |

### Checklist for raw pointer audit

When you encounter `new` anywhere in this codebase:

1. Is there a matching `delete` on **every** control-flow path (including exceptions)?
2. Is the pointer stored in a container? If so, does the container's destructor trigger `delete`?
3. Could the function throw between `new` and `delete`? → leak.
4. Is there a simpler value-type alternative?

## 3. Null Pointer Safety

### `std::getenv` returns `nullptr`

`std::getenv` returns `nullptr` (not an empty string) when the variable is unset. Dereference of this pointer is UB.

**Danger zones in this project**:

```cpp
// example.cpp — direct stream insertion of possible nullptr
std::cout << std::getenv("DATABASE_USERNAME");  // UB if not set

// dotenv.h — getenv wrapper handles null correctly:
inline std::string dotenv::getenv(const char* name, const std::string& def) {
    const char* str = std::getenv(name);
    return str ? std::string(str) : def;   // safe
}
```

**Rule**: Always use `dotenv::getenv(name, default)` inside this project. Never call `std::getenv` directly in application code.

### Checking null in resolve_vars

`dotenv::resolve_vars` already checks before dereferencing:

```cpp
if (const char* env_str = std::getenv(env_var.c_str()))
{
    resolved += env_str;   // safe — only inside the if
}
```

This pattern is correct. Use it as the model for any new `std::getenv` calls.

## 4. Ownership Transfer Patterns

### Return by value (NRVO/move)

The project's functions return `std::vector<SalesRecord>` and similar containers by value. Modern compilers apply Named Return Value Optimisation (NRVO) or at worst move semantics, so this is efficient.

**Do not** add `std::move` to a named local on the `return` statement — it suppresses NRVO:

```cpp
// BAD — disables NRVO
return std::move(result);

// GOOD — allows NRVO
return result;
```

### std::string short-string optimisation (SSO)

`SalesRecord` fields (`order_id`, `product_id`) are short strings. SSO means they rarely heap-allocate. Nevertheless, pass `SalesRecord` by `const&` in loops to avoid the copy cost of filling the SSO buffer repeatedly.

## 1. Unnecessary Heap Allocations

### New/delete round-trip for temporaries

The most impactful pattern in this codebase:

```cpp
// SLOW — two heap operations + extra copy
std::vector<T*> temp;
temp.push_back(new T(value));   // alloc + copy-construct

```cpp
// FAST — zero heap operations, one in-place construction
result.emplace_back(value);
```

**Rule of thumb**: if you see `new T(...)` where T is a plain struct (no virtual dispatch, no polymorphism), replace with value semantics.

When the final size is predictable, call `reserve` to avoid repeated reallocations:

```cpp
result.reserve(sales.size());   // worst case: all sales pass the filter
for (const auto& sale : sales) {
    if (product_set.count(sale.product_id)) {
        result.push_back(sale);
    }
}
```

### Use `emplace_back` over `push_back` for constructed-in-place objects

If building a struct field-by-field in a loop, `emplace_back` avoids a move:

```cpp
result.emplace_back(order_id, product_id, amount);   // constructs in-place
```

### `unordered_set` vs `set` for membership tests

The `join_with_inventory` function correctly uses `std::unordered_set` for O(1) average lookup:

```cpp
std::unordered_set<std::string> product_set;
// ...
if (product_set.count(sale.product_id)) { ... }
```

This is the right choice. A sorted `std::set` would be O(log n); a linear scan would be O(n). Verify this pattern is preserved in any refactoring.

### Prefer `contains` (C++20) over `count` for readability

```cpp
// C++17 — correct but count implies "how many"
if (product_set.count(sale.product_id)) { ... }

// C++20 — intent is clearer
if (product_set.contains(sale.product_id)) { ... }
```

## 3. String Operations

### Two `std::string::find` calls vs one

`find_var_start` in `dotenv.h` runs two `find` passes:

```cpp
size_t p1 = str.find('$', pos);
size_t p2 = str.find("${", pos);
```

Since `${` always starts with `$`, `p1 <= p2`. A single find is sufficient (see `known-hotspots.md §5`).

### `std::string_view` for read-only string parameters

Functions that only read a string and don't need ownership should prefer `std::string_view` (C++17) over `const std::string&`. This avoids an implicit conversion + allocation when a string literal is passed:

```cpp
// Current — forces an allocation if caller passes a string literal
static std::string strip_quotes(const std::string& str);

// Better — no allocation for string literals
static std::string strip_quotes(std::string_view str);
```

Note: changing the return type requires care; only do this if the callers are also updated.

### `std::ostringstream` for format_json / format_csv_output

The current use of `std::ostringstream` for small JSON/CSV strings is fine given the expected output size. No optimisation needed here.
