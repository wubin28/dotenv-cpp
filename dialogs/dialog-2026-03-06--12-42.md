## 方案A：日志文件分析与统计 CLI 工具（`log-analyzer`）

### 功能描述

用 dotenv 管理日志文件路径（`LOG_FILE`）、关键词过滤（`FILTER_KEYWORD=ERROR`）、时间窗口（`START_TIME`、`END_TIME`）、Top-N 统计（`TOP_N=10`）、输出格式（`OUTPUT_FORMAT=table`），在终端解析纯文本日志文件，按关键词统计频率、按时间段筛选、输出摘要报告。

### 人类程序员容易犯的 C++ Bug

| Bug 类型 | 具体表现 |
|---|---|
| **`npos` 未检查导致崩溃** | `line.substr(line.find("ERROR"))` 当 `find()` 返回 `npos` 时，`substr(npos)` 行为未定义 |
| **字符串拼接 O(N²) 性能问题** | 在循环里用 `report += line + "\n"` 拼接百万行日志，每次拼接都复制整个字符串，应用 `std::ostringstream` |
| **整数溢出** | 用 `int total_bytes` 统计日志文件大小，超过 2GB 文件变负数，应用 `size_t` 或 `int64_t` |
| **悬空引用** | `const char* keyword = dotenv::getenv("FILTER_KEYWORD").c_str()`，`std::string` 临时对象析构后 `keyword` 成悬空指针 |
| **大文件 OOM** | `std::vector<std::string> all_lines` 将 500MB 日志全量读入内存崩溃，应流式逐行处理 |
| **`stoi` 转换未防御** | `int top_n = std::stoi(dotenv::getenv("TOP_N",""))` 当 `.env` 缺失该键时传入空串，`stoi` 抛 `std::invalid_argument` |

### 特点

- dotenv 的使用极其自然：**日志路径、过滤参数不应硬编码**，和 dotenv 核心价值完全契合
- 业务逻辑学员零成本理解（"从日志里找 ERROR 行" 无需任何领域知识）
- `npos` 未检查 + 字符串拼接性能 是 C++ 初学者最高频的两类 bug，在同一段代码里自然共存

### 优势

- **入门门槛极低**：只需要 `std::string`、`getline`、`map` 的基础知识
- **可感知的性能差异**：`string +=` 改为 `ostringstream` 后，处理 10 万行日志的耗时从秒级降到毫秒级，AI 优化效果可以用 `std::chrono` 直接量化
- **AI 辅助场景丰富**：让 AI 自动生成测试日志文件、识别热路径、将 `npos` 检查模式提取为 helper 函数

### 劣势

- 日志格式多样（Nginx、Java、syslog），如果未统一格式规范，学员可能花时间在格式解析而非 bug 修复
- 建议在需求描述中规定一种固定格式（如 `[YYYY-MM-DD HH:MM:SS] LEVEL: message`）来规避这一问题

### 适用场景

非常适合作为**CSV 报表工具的平行练习作业**，两者使用相同的核心 bug 模式（字符串拼接性能、越界、类型转换）但场景不同，学员可以巩固相同知识点而不觉得重复。

## 方案B：学生成绩批处理与报告工具（`grade-reporter`）

### 功能描述

用 dotenv 管理成绩文件路径（`GRADES_FILE`）、及格线（`PASSING_SCORE=60`）、各等级分界线（`GRADE_A=90`、`GRADE_B=75`、`GRADE_C=60`）、班级名称（`CLASS_NAME`）、报告输出路径（`REPORT_FILE`），在终端读取 TXT/CSV 成绩文件，计算均值/最高/最低分、统计各等级人数、生成按分数排名的报告并写入文件。

### 人类程序员容易犯的 C++ Bug

| Bug 类型 | 具体表现 |
|---|---|
| **浮点比较精度 bug（最隐蔽）** | `if (score >= 90.0)` 判断等级，但 `score` 由 `stod("90")` 解析后可能是 `89.99999...`，导致 90 分被划为 B 等而非 A 等，结果静默出错而非崩溃 |
| **`vector` 越界 UB** | 解析成绩行 `fields[2]`（期望第 3 列是分数），当某行只有 2 列时 `operator[]` 越界，行为未定义 |
| **`char` 数组缓冲区溢出** | `char name[16]` 用 `strcpy(name, student_name.c_str())` 存储姓名，输入超过 15 字符时栈溢出 |
| **内存泄漏** | `Student* list = new Student[n]` 手动分配学生数组，中途 `return` 的错误分支未 `delete[]` |
| **整数截断** | 从 dotenv 读取 `PASSING_SCORE`，用 `int passing = std::stoi(...)` 存储，再与 `double score` 比较时隐式转换导致 `59.9 >= 60` 边界判断出错 |
| **排序谓词违反严格弱序** | 自定义 `sort` 比较器写成 `a.score >= b.score`（含等号），违反严格弱序，触发 UB |

### 特点

- **浮点精度 bug 是该方案的核心亮点**：`90.0` 分被划为 B 等，结果完全错误但程序不崩溃，需要 AI 辅助调试工具（打印中间值、对比预期结果）才能发现
- 业务逻辑零门槛：成绩计算和等级划分人人都能看懂
- 与 dotenv 的耦合自然：**不同课程/考试的分界线不同，放在 `.env` 里切换极其合理**

### 优势

- **浮点精度 bug 教学价值极高**：这类 bug 在生产环境中极常见（金融计算、科学计算），且 AI 辅助发现此类 bug 的效果显著——可以让学员先自己找不出来，再用 AI 辅助分析
- **Bug 类型多样**：同时覆盖内存（`new`/`delete[]`）、越界（`vector[]`）、精度（`double` 比较）、安全（`strcpy` 溢出），四种经典 C++ 问题一次覆盖
- **测试数据 AI 生成极方便**：让 AI 生成包含边界分数（60、90、100）的测试数据集，验证 bug 的复现

### 劣势

- 浮点精度问题需要老师提前铺垫一下"为什么 `stod("90") != 90.0`"的背景，否则学员容易迷失
- 功能本身较简单，规模感不如 CSV 多源 Join 强

### 适用场景

特别适合用来**专项训练 AI 辅助调试隐蔽 bug** 的场景（结果错误但程序不崩溃）。可以设计成：给学员一个"有 bug 但能运行"的程序，让他们用 AI 工具逐步排查为什么 5 个 90 分学生被报告为 B 等。
